\begin{abstract}

    This work presents a systematic, reproducible study of cache hierarchy
    effects on a set of five PolyBench kernels using gem5. We perform a
    controlled parameter sweep over cache sizes, cache line sizes, and
    associativities, and collected detailed gem5 statistics for each run. We
    focus on standard microarchitectural metrics (miss rate, \textsc{mpki},
    \textsc{ipc}/\textsc{cpi}, memory-related stall cycles) and on relative
    speedup against a baseline configuration. Results show workload-dependent
    tradeoffs: some kernels are dominated by data working‑set effects (L1D/L2),
    while others exhibit negligible icache pressure; visualization and
    \textsc{mpki} decomposition reveal where increased capacity or associativity
    yields diminishing returns. The artifacts and methodology aim to help
    students and researchers perform rigorous, replicable cache design‑space
    evaluations.

\end{abstract}

\section{Introduction}
\subsection{Context}

When designing or modeling computer architectures, memory hierarchy
configurations are responsible for much of a system's performance across a
variety of workloads. Modern computer architectures face a fundamental problem,
known as a \textit{memory wall} \cite{wulf:1995}, the
growing disparity between the speed of \textsc{cpu} cores and main memory
latencies. To mitigate this, most if not all serious implementations of modern
instruction set architectures (\textsc{isa}s) employ complex memory hierarchies,
split across multiple levels, and which exploit locality. This hierarchy
encompasses each core's register file, different cache levels, as well as
main and secondary memories, and is one of the cornerstones of contemporary
architecture design. Its configuration parameters determine not only the maximum
throughput of high-performance systems, but also the energy efficiency and the
viability of low-power systems, as in single-board computers (\textsc{sbc}s) and
distributed systems.

The efficacy of a given memory hierarchy hinges on the ability to exploit
temporal and spatial localities in data accesses. However, the precise
configuration of memory hierarchy parameters---be they cache sizes,
associativity, latencies, replacement policies, block sizes, \textit{etc}.---to
achieve optimal throughput for a given workload remains a complex and
domain-dependent question, and requires detailed empirical exploration.

\subsection{Motivation}

Although much research has been done on the impact of cache configuration
parameters on a variety of workloads, most analyses measure only specific
individual metrics, focus on a small domain of specific workloads, or evaluate a
limited set of configuration parameters in isolation. Thus, it is often
difficult to systematically compare the relative effects of different cache
design choices on a wide array of workloads in a controlled, reproducible
manner. The lack of such multi-parameter evaluations motivates the need for a
methodology that enables consistent and interpretable explorations of the design
space.

\subsection{Objective}

In this work, we aim to conduct controlled architectural experiments using
select kernels from the PolyBench benchmark suite \cite{polybench:2016} in order
to investigate the impact of different cache memory configurations on select
performance metrics. A set of experimental scenarios is designed by sweeping key
parameters of the memory hierarchy, including cache size, cache line size, and
associativity levels for all cache levels. The experiments will be conducted
using the gem5 architectural simulator \cite{gem5-20:2007}. The evaluation is
based on the analysis of select performance metrics such as multi-level miss
rates and \textsc{mpki} (Misses Per Kilo Instructions) counts, speedup across
different simulated architectures, and memory-related stall cycles. Through this
analysis, we seek to provide a deeper understanding of how variations in cache
parameters effect changes in standard microarchitectural metrics in the
execution behavior of PolyBench benchmarks, and thus enable the identification
of potential optimization opportunities within the memory hierarchy.

\subsection{Contribution}

We provide a systematic and reproducible evaluation of cache hierarchy
parameters across an extensible set of benchmark kernels, using the gem5
simulator. The main contributions are threefold. First, we perform a controlled
multi-parameter exploration of cache line size, associativity, and capacity,
enabling direct comparison of their relative impact on key microarchitectural
metrics such as miss rates, \textsc{mpki}, and \textsc{ipc}/\textsc{cpi}.
Second, we develop an automated simulation and data collection framework that
ensures consistency and reproducibility across a large number of experimental
configurations. Third, we provide a cross-workload analysis that highlights how
different access patterns and dataset sizes influence the sensitivity of
performance to cache design choices, offering practical insights into the
trade-offs involved in memory hierarchy configuration.

\section{Related Work}

From an architectural optimization perspective, Khoshavi and DeMara
\cite{khoshavi:2018} investigate the use of emerging non-volatile memories
within the cache hierarchy. The authors propose the \textsc{rrap} strategy (Read
Reference Activity Persistent), a hybrid architecture that replaces conventional
\textsc{sram} with a combination of low- and high-retention
\textsc{stt}-\textsc{ram} (Spin-Transfer Torque Random Access Memory). Their
results indicate a 51.7\% reduction in L2 cache read miss rate, along with
improvements in area and energy consumption, without degrading system
performance.

From a methodological standpoint, Bueno et al. \cite{bueno:2024} emphasize the
importance of experimental rigor and reproducibility in memory architecture
evaluation. Their work proposes a weight redefinition methodology for simulation
intervals based on memory activity (\textsc{mpki}). The authors demonstrate that
this approach reduces discrepancies in the analysis of replacement policies by
more than 30\%, highlighting that the reliability of results strongly depends on
the quality of the experimental procedure.

The validation of proposals in this domain relies on well-established simulation
infrastructures. The gem5 architectural simulator \cite{gem5-20:2007} has become
a fundamental tool for detailed architectural modeling, enabling systematic
exploration of design parameters. In parallel, the investigation of spatial and
temporal locality requires representative workloads, for which the PolyBench
suite \cite{polybench:2016} is widely adopted due to its collection of
mathematical kernels. The versatility of gem5 has further driven recent research
focused on Design Space Exploration (\textsc{dse}), of which a representative
example is the work of Shahid et al. \cite{shahid:2025}, in which the authors
employ the framework to improve the performance of the \textsc{risc}-V
\textsc{cva6} core. Through parametric simulations, they identify configurations
capable of balancing hardware cost and processor performance. This approach
reinforces the suitability of gem5 not only as a tool for evaluating traditional
cache designs but also as a key instrument for the rigorous validation of
emerging and extensible architectures.

\section{Methodology}
\subsection{Testing Strategy with gem5}
\subsubsection{General Approach Description}

Tests were conducted using the gem5 simulator, enabling precise simulation of cache behavior across different hardware configurations. All workloads were compiled for x86 instruction set architecture, ensuring compatibility with the gem5 x86 simulator backend. For each workload, 32 distinct simulations were performed with varying cache configurations, enabling extensive analysis of cache parameters' impact on performance.

\subsubsection{Automation Framework (run\_all\_simulations.py)}

The \texttt{run\_all\_simulations.py} script serves as the central orchestrator for simulations, implementing the following features:

\begin{itemize}
  \item \textbf{Parallel Execution}: Uses \texttt{multiprocessing} to distribute simulations across multiple processors, significantly reducing total execution time
  \item \textbf{Core Affinity}: Implements \textsc{cpu} pinning to improve data locality and reduce measurement variability
  \item \textbf{Output Synchronization}: Ensures logs from different processes do not mix during parallel execution
  \item \textbf{Completion Verification}: Automatically detects successful simulations by the presence of \texttt{stats.txt} with critical metrics (\texttt{simTicks}, \texttt{system.cpu.numCycles})
\end{itemize}

\subsubsection{Additional Cache Parameters}

In addition to size variations, the following configurations were tested:

\begin{itemize}
  \item \textbf{Cache Line Size}: 32, 64, 128, 256 bytes
  \item \textbf{L1/L2 Associativity}: 1, 2, 4, 8, 16 ways
\end{itemize}

All cache dimensions and associativity values were rounded to powers of 2, as required by the gem5 simulator.

Totaling 32 different variations per workload (13 base configurations + cache line variations + associativity variations).

\subsubsection{Statistics Collection}

Each simulation generates a \texttt{stats.txt} file containing:

\begin{itemize}
  \item \textbf{\textsc{cpu} Cycles}: Total number of simulation cycles
  \item \textbf{Cache Hits/Misses}: Hit and miss rates at each cache level
  \item \textbf{Latencies}: Average access time at different levels
  \item \textbf{Throughput}: Instructions per cycle (\textsc{ipc})
  \item \textbf{Memory Pressure}: System bandwidth utilization
\end{itemize}

Data are collected through gem5-specific calls:
\begin{small}
\begin{verbatim}
m5_reset_stats(0, 0);   // Reset counters
// Code of interest here
m5_dump_stats(0, 0);    // Capture statistics
\end{verbatim}
\end{small}

\subsection{Workload Description}\label{sub:workloads}

\subsubsection{\textsc{gemm} (General Matrix Multiply - PolyBench)}

\paragraph{Type}: Linear Algebra | \paragraph{Access Pattern}: Sequential with Tiling

\paragraph{Description}

\textsc{blas}-level implementation of generalized matrix multiplication: $C := \alpha \cdot A \cdot B + \beta \cdot C$. Uses parameterized datasets from the PolyBench benchmark.

\paragraph{Kernel Main}

\begin{small}
\begin{verbatim}
for (i = 0; i < _PB_NI; i++) {
  for (j = 0; j < _PB_NJ; j++)
    C[i][j] *= beta;
  for (k = 0; k < _PB_NK; k++) {
    for (j = 0; j < _PB_NJ; j++)
      C[i][j] += alpha * A[i][k] *
                 B[k][j];
  }
}
\end{verbatim}
\end{small}

\paragraph{Dimensions Used (MEDIUM\_DATASET)}

\begin{itemize}
  \item $NI = 200$ (rows of A)
  \item $NJ = 220$ (columns of B)
  \item $NK = 240$ (shared dimension)
\end{itemize}

\paragraph{Performance Characteristics}

\begin{itemize}
  \item \textbf{Temporal Locality}: Very High (C[i][j] reused multiple times)
  \item \textbf{Spatial Locality}: High (access in tiling-friendly pattern)
  \item \textbf{Cache Hit Rate}: Expected high, especially with cache blocking
  \item \textbf{Cache Sensitivity}: High - optimized configurations can double performance
\end{itemize}

\paragraph{Mathematical Equation}

\[C_{ij} = \beta \cdot C_{ij} + \alpha \sum_{k=0}^{NK-1} A_{ik} \cdot B_{kj}\]

\paragraph{Complexity}

\begin{itemize}
  \item \textbf{Temporal}: $O(NI \times NJ \times NK) = O(200 \times 220 \times 240) = O(10.56 \times 10^{6})$ operations
  \item \textbf{Spatial}: $O(NI \times NK + NK \times NJ + NI \times NJ) \approx 1.1$ MB with double precision
\end{itemize}

\subsubsection{\textsc{ata}x (Matrix Transpose Vector - PolyBench)}

\paragraph{Type}: Linear Algebra | \paragraph{Access Pattern}: Sequential with Transposition

\paragraph{Description}

Computes matrix transpose application followed by vector multiplication: $y := A^T \cdot (A \cdot x) + z$. Demonstrates non-sequential access patterns in one dimension.

\paragraph{Kernel Main}

\begin{small}
\begin{verbatim}
for (i = 0; i < _PB_N; i++)
  for (j = 0; j < _PB_M; j++)
    y[j] += A[i][j] * x[i];

for (i = 0; i < _PB_N; i++)
  for (j = 0; j < _PB_M; j++)
    z[i] += A[i][j] * y[j];
\end{verbatim}
\end{small}

\paragraph{Dimensions Used (LARGE\_DATASET)}

\begin{itemize}
  \item $M = 1900$ (number of columns)
  \item $N = 2100$ (number of rows)
\end{itemize}

\paragraph{Performance Characteristics}

\begin{itemize}
  \item \textbf{Temporal Locality}: Moderate (y and z redefinition in inner loops)
  \item \textbf{Spatial Locality}: Moderate-High (column access in second stage)
  \item \textbf{Cache Hit Rate}: Moderate - non-sequential access hurts cache
  \item \textbf{Cache Sensitivity}: Moderate
\end{itemize}

\paragraph{Mathematical Equations}

\[y_j = \sum_{i=0}^{N-1} A_{ij} \cdot x_i\]
\[z_i = z_i + \sum_{j=0}^{M-1} A_{ij} \cdot y_j\]

\subsubsection{Floyd-Warshall (Shortest Path - PolyBench)}

\paragraph{Type}: Classic Algorithm | \paragraph{Access Pattern}: Complex Reuse

\paragraph{Description}

Classic dynamic programming algorithm for finding shortest paths between all pairs of vertices. Demonstrates complex data reuse pattern with three levels of nesting.

\paragraph{Kernel Main}

\begin{small}
\begin{verbatim}
for (k = 0; k < _PB_N; k++) {
  for (i = 0; i < _PB_N; i++)
    for (j = 0; j < _PB_N; j++)
      path[i][j] = path[i][j] <
        path[i][k] + path[k][j] ?
        path[i][j] :
        path[i][k] + path[k][j];
}
\end{verbatim}
\end{small}

\paragraph{Dimensions Used (SMALL\_DATASET)}

\begin{itemize}
  \item $N = 180$ (number of nodes)
  \item Distance matrix: $180 \times 180$ elements
\end{itemize}

\paragraph{Performance Characteristics}

\begin{itemize}
  \item \textbf{Temporal Locality}: Very High (path[i][j], path[i][k], path[k][j] reused)
  \item \textbf{Spatial Locality}: High (access in dense 2D pattern)
  \item \textbf{Cache Hit Rate}: High - working set fits in L3
  \item \textbf{Cache Sensitivity}: High - L3 miss is very costly
  \item \textbf{Access Pattern}: Streaming with reuse (layer-wise)
\end{itemize}

\paragraph{Complexity}

\begin{itemize}
  \item \textbf{Temporal}: $O(N^3) = O(180^3) \approx 5.8 \times 10^{6}$ comparisons
  \item \textbf{Spatial}: $O(N^2) \approx 129.6$ \textsc{kb} for distance matrix (integers)
\end{itemize}

\paragraph{Applications}

Network routing, graph analysis, path length algorithms.

\subsubsection{Jacobi-2D (Iterative Method - PolyBench)}

\paragraph{Type}: Numerical Simulation | \paragraph{Access Pattern}: 2D Stencil

\paragraph{Description}

Implementation of the Jacobi iterative method for solving linear systems in 2D. Uses ``stencil'' access pattern where each point depends on its 5 neighbors (center, top, bottom, left, right).

\paragraph{Kernel Main}

\begin{small}
\begin{verbatim}
for (t = 0; t < _PB_TSTEPS; t++) {
  for (i = 1; i < _PB_N-1; i++)
    for (j = 1; j < _PB_N-1; j++)
      B[i][j] = 0.2 * (A[i][j] +
        A[i][j-1] + A[i][j+1] +
        A[i+1][j] + A[i-1][j]);
  for (i = 1; i < _PB_N-1; i++)
    for (j = 1; j < _PB_N-1; j++)
      A[i][j] = 0.2 * (B[i][j] +
        B[i][j-1] + B[i][j+1] +
        B[i+1][j] + B[i-1][j]);
}
\end{verbatim}
\end{small}

\paragraph{Dimensions Used (MEDIUM\_DATASET)}

\begin{itemize}
  \item Mesh: $250 \times 250$
  \item Time iterations (TSTEPS): 100
  \item Total operations: $\sim 6.25 \times 10^{6}$
\end{itemize}

\paragraph{Performance Characteristics}

\begin{itemize}
  \item \textbf{Temporal Locality}: Moderate-High (alternation between A and B)
  \item \textbf{Spatial Locality}: Very High (access to 5 neighbors close together)
  \item \textbf{Cache Hit Rate}: Very High - stencil fits in cache
  \item \textbf{Cache Sensitivity}: Moderate (well-defined working set)
  \item \textbf{Access Pattern}: Stencil with regular reuse
\end{itemize}

\paragraph{Computational Equation}

\begin{multline*}
B[i,j]^{(n)} = 0.2 \times (A[i,j]^{(n)} + A[i,j-1]^{(n)} + A[i,j+1]^{(n)} \\
+ A[i+1,j]^{(n)} + A[i-1,j]^{(n)})
\end{multline*}


\paragraph{Applications}

Heat transfer simulation, partial differential equation solving, computational fluid dynamics.

\subsubsection{Seidel-2D (Gauss-Seidel - PolyBench)}

\paragraph{Type}: Numerical Simulation | \paragraph{Access Pattern}: 2D Stencil with Convergence

\paragraph{Description}

Variant of the Jacobi method using Gauss-Seidel sweep, where each point uses already-updated values in the current iteration. Presents slight reduction in locality compared to Jacobi.

\paragraph{Kernel Main}

\begin{small}
\begin{verbatim}
for (t = 0; t <= _PB_TSTEPS-1; t++)
  for (i = 1; i <= _PB_N-2; i++)
    for (j = 1; j <= _PB_N-2; j++)
      A[i][j] = (A[i-1][j-1] +
        A[i-1][j] + A[i-1][j+1] +
        A[i][j-1] + A[i][j] +
        A[i][j+1] + A[i+1][j-1] +
        A[i+1][j] + A[i+1][j+1])
        / 9.0;
\end{verbatim}
\end{small}

\paragraph{Dimensions Used (MEDIUM\_DATASET)}

\begin{itemize}
  \item Mesh: $400 \times 400$
  \item Time iterations (TSTEPS): 100
  \item Neighborhood pattern: 9 points (Moore neighborhood)
\end{itemize}

\paragraph{Performance Characteristics}

\begin{itemize}
  \item \textbf{Temporal Locality}: Moderate (limited reuse between timesteps)
  \item \textbf{Spatial Locality}: Very High (9 neighbors accessed sequentially)
  \item \textbf{Cache Hit Rate}: Very High - better than Jacobi in some aspects
  \item \textbf{Cache Sensitivity}: Moderate-High
  \item \textbf{Access Pattern}: Dense stencil with 9 points
\end{itemize}

\paragraph{Computational Equation}

\[A[i,j]^{(n+1)} = \frac{1}{9}\sum_{p=-1}^{1}\sum_{q=-1}^{1} A[i+p,j+q]^{(n)}\]

\paragraph{Difference with Jacobi}

Uses 9 points in Moore neighborhood versus 5 points in von Neumann neighborhood, resulting in larger data volume accessed but possibly better convergence.

\section{Results}
\subsection{Overview}

Overall, only a small subset of the values explored in our parameter sweep had a
significant impact on the sampled metrics. The most dominant parameter across
all workloads by far was the cache line size, wherein doubling the line size
tends to significantly reduce L1D miss rate, often approaching a halving for
highly sequential access patterns (\textsc{ipc} improves accordingly), probably because
larger lines increase spatial locality utilization per miss, and associativity
effects are mostly tied to cache entry conflicts within a set. As such,
associativity effects were largely flat, and the capacity presets impact
performance selectively, and only when dataset footprints\footnote{To be
understood as the kernel's hot working set.} fit in the larger caches.

\subsection{Cache Line Size Effects}\label{sub:cacheline}

Larger lines better exploit spatial locality by coalescing contiguous words per
tag. Thus, fewer L1D misses propagate as lower miss latencies and lead to \textsc{cpi}
gains. To point out some examples, Fig.~\ref{fig:atax-speedup-line} shows that,
for the \textsc{ata}x kernel, contiguous accesses give high spatial locality, and
doubling line size captures more contiguous work per refill, which in turn
reduces the number of refills. On Fig.~\ref{fig:jacobi-mpki-line}, Jacobi-2D
shows the same spatial-locality response, and on
Fig.~\ref{fig:seidel-stalls-line}, we see that Seidel-2D sees dramatic
reductions in miss latencies with larger lines.\footnote{In these latter two,
it's worth noting that \textsc{cpi} gains are present but smaller, because a significant
fraction of execution is compute. A larger dataset would probably correspond to
a more dramatic \textsc{cpi} reduction.} Finally, Fig.~\ref{fig:gemm-cpi-line} shows
modest \textsc{cpi} improvements for \textsc{gemm}, arguably due to inherent temporal locality and
compute intensity.

\subsection{Associativity Changes Least Significant}

Cache associativity was by far the least significant cache parameter, and the
impact was almost negligible (a few tenths of a percent). Mechanically,
associativity reduces conflict misses by allowing multiple cache lines that map
to the same set to coexist. In our kernels, most accesses are regular and spread
across many sets. That is because strides are low or simple stencils are used.
Thus, conflict pressure is low, suggesting that misses are dominated by
compulsory and capacity effects. The only exception is when associativity is one
(the cache is directly mapped), due to the fact that \textit{all} conflicts will
lead to an eviction. This can be seen in Fig.~\ref{fig:comp-ipc-l1} as slightly
lower instructions per cycle when the L1 caches are directly mapped.

    \centering

\end{figure}

\subsection{Capacity Presets}

The presets change the nominal capacity of L1, L2, and L3. Larger overall
capacities reduce misses \textit{when} the kernel's hot working set is small
enough that the last-level cache (\textsc{llc}) can hold a significant fraction of the
set. When the set is significantly larger than the \textsc{llc} can hold, or when most
misses are compulsory, such that spatial locality optimizations (e.g., larger
lines) are more impactful, as we exemplified in \ref{sub:cacheline}, capacity
changes have little effect. Fig.~\ref{fig:comp-ipc-size} shows which workloads
had most gains, and Fig.~\ref{fig:atax-speed-size} shows the effect on the \textsc{ata}x
kernel, which had the most dramatic improvement.

\subsection{Some Observations}

\subsubsection{Instruction cache misses} Across all 155 simulations, the total
miss count for the instruction cache was \textit{always} zero. This is to be
expected, because by the time the workload binary calls the \textit{libm5} \textsc{api}
to start counting statistics, the working set of instructions has already been
fetched into the instruction cache and easily fits within its capacity: the
relocatable object corresponding to the main translation unit of each workload
is on the order of a few kilobytes.

\subsubsection{Impact of dataset sizes} As we detailed in \ref{sub:workloads},
different workloads were set to use different dataset size configurations due to
time constraints. The \textsc{ata}x kernel was compiled with the LARGE\_DATASET
definition, the Floyd-Warshall kernel with SMALL\_DATASET, and the remaining
kernels with MEDIUM\_DATASET. This may explain why the \textsc{ata}x simulations
exhibited the largest absolute jumps in \textsc{mpki}, miss rates, total
latencies, and \textsc{cpi}, whereas Floyd-Warshall exhibited the smallest jumps: larger
dataset sizes may increase working-set pressure, such that, if the working set
is near the cache size boundary, changes in total cache capacity and line size
may materially change miss behavior.
